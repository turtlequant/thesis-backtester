# 从回测到生产：三层实时分析架构

## 背景

当前框架以固定截面（每半年）做回测验证，已证明 alpha 有效（+7.1pp over CSI300）。但实际投资中，估值修复不按日历走——股价每天变，买入时机是动态的。

核心矛盾：**Agent 分析的是基本面（季度变化），但买入时机取决于价格（每天变化）。**

## 三层架构

```
第一层 (季度)           第二层 (每日)            第三层 (触发时)
财报驱动评估        →   价格信号监控        →   定向资讯校验
                              │
解决"买不买"            解决"什么时候买"         解决"有没有新风险"

Agent 6章深度分析       纯数值比较               搜索最近公告/新闻
跟财报披露周期走         成本为零                 仅在行动前触发
¥0.4/只                 每天自动跑               可手动，后续自动化
```

### 第一层：财报驱动的完整评估

**是什么**：和现在 Agent 分析做的事一样——六章结构化分析、评分、买入/观望/回避建议。

**和当前的区别**：触发时机从"固定半年截面"改为"跟财报披露周期走"。

A 股财报披露节奏：
- 4 月 30 日前：年报 + 一季报
- 8 月 31 日前：半年报
- 10 月 31 日前：三季报

自然截面日期：5 月初、9 月初、11 月初（财报季结束后立刻跑）。每年 3 次有效评估，比固定半年截面更贴合信息更新的实际节奏。

**产出**：
- 候选名单（通过量化筛选的股票池）
- 每只候选的深度分析报告
- 关键价格锚点：正常价、买入价、卖出价（来自苹果买卖模型）

**不变的原则**：
- 不看股价，只看基本面和行业条件
- 盲测协议照旧
- 时间边界照旧

### 第二层：价格信号监控

**是什么**：每天拉最新行情，和第一层产出的价格锚点对比。

**监控信号**：
- 当前价 < Agent 买入价 → 进入买入区间
- 当前价 > Agent 卖出价 → 考虑止盈
- PE/PB/股息率穿越筛选阈值 → 新股票进入/离开候选池
- 相对 Agent 分析时的价格变动幅度（涨/跌超过 20% 时标记）

**实现成本**：极低。不调用 LLM，纯数值比较。每天自动跑一次脚本，耗时秒级。

**产出**：
- 每日信号列表（哪些股票进入/离开买入区间）
- 异常标记（价格大幅偏离 Agent 预期时提醒复查）

**关键约束**：第二层只管价格信号，不做基本面判断。如果股价因暴雷暴跌到买入价以下，第二层会误报"进入买入区间"——这由第三层防御。

### 第三层：定向资讯校验

**是什么**：在第二层触发买入信号时，做一次定向的资讯检查，确认没有第一层分析时未知的新风险。

**触发条件**：第二层产生买入信号时（不是每天跑，是按需触发）。

**检查内容**：
- 该股票最近 30 天的公告（重大事项、预亏预减、ST 风险）
- 最近的新闻舆情（暴雷、被调查、高管变动等）
- 行业层面的重大变化

**实现思路**：
- MVP：手动检查（触发时自己看一眼东方财富/巨潮资讯）
- V1：自动拉取巨潮公告 API + 简单关键词过滤
- V2：LLM 摘要最近公告，判断是否影响投资逻辑

**产出**：
- 通过 → 维持 Agent 原有买入建议
- 发现重大负面 → 标记"暂缓"，等下一轮第一层重新评估

## 三层协作流程

```
财报季结束 (5月/9月/11月)
    │
    ▼
第一层: screen + agent → 产出候选名单 + 买卖价格
    │
    ▼ (价格锚点写入监控列表)
每日运行
    │
    ▼
第二层: 拉行情 → 对比买卖价格
    │
    ├── 无信号 → 继续监控
    │
    └── 触发买入信号
            │
            ▼
        第三层: 查最近公告/新闻
            │
            ├── 无异常 → 确认买入
            │
            └── 发现风险 → 暂缓，等下轮评估
```

## 与回测框架的关系

三层架构是回测框架的自然延伸，不是替代：

| | 回测模式 | 生产模式 |
|--|---------|---------|
| 第一层触发 | 固定截面日期 | 财报披露后 |
| 第二层 | 不存在（事后采集收益） | 每日价格监控 |
| 第三层 | 不存在 | 定向资讯校验 |
| 目的 | 验证方法论有效性 | 实际投资决策 |

回测证明了第一层的 alpha 有效（+7.1pp），生产模式加上第二层和第三层是为了把这个 alpha 转化为可操作的买卖时机。

## 实施优先级

| 步骤 | 内容 | 依赖 | 工作量 |
|------|------|------|--------|
| 0 | 等市场反馈，确认方向 | 文章传播效果 | — |
| 1 | 第二层价格监控 | 从 structured.json 提取买卖价 | 半天 |
| 2 | 第一层改为财报驱动 | 调整截面日期逻辑 | 1 天 |
| 3 | 第三层 MVP（手动） | 无 | 0 |
| 4 | 第三层 V1（公告 API） | 巨潮/东财接口 | 2-3 天 |

## 生产模式：指数成分股策略

回测模式需要"大海捞鱼"（全市场 5000+ 只筛选），但生产模式不需要。锁定一个指数成分股池，全量 Agent 评估即可。

### 为什么选指数成分股

- 流动性好，不存在买不进卖不出的问题
- 成分股定期调整，自带优胜劣汰
- 信息充分，财报披露及时
- 机构研究覆盖度高，AI 分析的财务数据更完整

### 三档可选

| 指数 | 成分股 | 每轮 Agent 成本 | 耗时(并发10) |
|------|--------|----------------|-------------|
| 中证50 | 50 | ¥20 | ~1 小时 |
| 沪深300 | 300 | ¥120 | ~5 小时 |
| 中证500 | 500 | ¥200 | ~8 小时 |

推荐从沪深300起步——覆盖面够广、成本可控。

### 运作方式

```
每季度财报季后 (5月/9月/11月)
    │
    ▼
300 只成分股全量 Agent 评估 (¥120, 5h)
    │
    ▼
按评分排序，取 Top 5-15 作为持仓组合
    │
    ▼
第二层每日监控：持仓价格 vs 买入价/卖出价
    │
    ├── 某只跌入加仓区 → 第三层校验 → 加仓
    ├── 某只涨到卖出价 → 减仓/止盈
    └── 新股进入 Top 15 / 旧股跌出 → 调仓
```

### 和回测模式的区别

| | 回测模式 | 生产模式 |
|--|---------|---------|
| 股票池 | 全市场 → 量化筛选 → Agent | 指数成分股 → 全量 Agent |
| 量化筛选层 | 必须（5000→50） | 不需要（池子就 300 只） |
| Agent 覆盖率 | 10/50 = 20% | 300/300 = 100% |
| 持仓管理 | 无（只出报告） | 维护 5-15 只实盘持仓 |
| 调仓频率 | 无 | 季度大调 + 每日微调 |
| 年化成本 | 一次性 ¥48 | 约 ¥360/年 (3次×¥120) |

### 持仓管理规则（待设计）

需要明确的规则：
- 初始建仓：Top N 等权 or 按评分加权？
- 单只上限：不超过总仓位的 X%？
- 调仓阈值：评分变动多少才触发调仓？
- 止损规则：跌破 Agent 预估的最大安全边际时强制卖出？

这些规则属于持仓管理算子（position_management）的延伸，可以配置在 strategy.yaml 中。
